ICLR 2027 target · proposal and specs

EditDecide: recasting instruction-based image editing as decision-making

This site presents the project proposal, benchmark design, JSON schema, method stack, experiment plan, and compute-constrained execution plan for EditDecide. The central claim is that trustworthy image editors should decide whether to edit, abstain, flag, ask, confirm, or request a mask before producing a plausible but wrong image.

Core formula

One benchmark for action selection

Same image pool

Shared bottom images support feasibility, self-knowledge, ambiguity, and human seed slices.

Same JSON schema

Every field must support a correct decision, an explanation, an experiment, or reproducibility.

Different decision tracks

S1, S1', S2, S3, S3', and R are connected through twins, cached outputs, and split IDs.

One decision benchmark

Systems are evaluated on whether they choose the right action before editing.

Unified action space

The system chooses among six actions

ActionWhen it should be selectedPrimary track
EDITThe request is feasible, clear enough, and expected risk is below the cost threshold.All
ABSTAINInformation is unrecoverable or the editor is likely to fail in a high-cost way.G1/G2
FLAGThe instruction contains a false premise, contradiction, policy issue, or impossible target.G1
ASKClarification has positive expected value after subtracting the user's cost.G3
CONFIRMThe likely intent is known but the edit is costly enough to require confirmation.G3
REQUEST_MASKLanguage is inefficient and spatial input is the correct disambiguation channel.G3
