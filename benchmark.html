EditDecide-Bench

Dataset and benchmark specification

EditDecide-Bench is one unified decision dataset, not separate feasibility, calibration, and clarification benchmarks. Its design uses a shared image pool, shared JSON schema, and multiple decision tracks.

Five Dataset Layers

Layer 1Raw episode

Image plus user instruction.

Layer 2Decision annotation

Feasibility, ambiguity, gold action, rationale, acceptable actions.

Layer 3Twin structure

Feasible twins and counterfactual intent twins.

Layer 4Editor behavior

Outputs, refusal text, generated images, hidden states, probe predictions.

Layer 5Evaluation and reproducibility

Judge labels, human labels, metric inputs, cached paths, split IDs.

Five Infeasibility Types

false_premise

The instruction refers to something absent from the image. Example: recolor a red sports car when no car is visible. Expected action: FLAG.

impossible

The requested information is unrecoverable, such as restoring an occluded face with the same identity. Expected action: ABSTAIN.

contradictory

The instruction contains mutually incompatible constraints. Expected action: FLAG.

ill_posed

The instruction needs missing external context, such as brand guidelines. Expected action: ASK or FLAG.

out_of_scope_or_unsafe

The edit is outside the project scope or safety scope. Expected action: FLAG.

Seven Ambiguity Types

target

Multiple possible objects. Often requires REQUEST_MASK or ASK.

operation

The operation is underspecified, such as "fix the background". Usually requires ASK.

region

The affected region is unclear, such as "make it brighter". May require language or mask input.

degree

Edit strength is unclear, such as "a little warmer". Often requires CONFIRM.

style

The style target is underspecified, such as "more cinematic". Usually requires ASK.

preservation

What must remain unchanged is unclear. Often requires CONFIRM.

interaction

Language is inefficient and spatial input is better, as in "remove that thing". Expected action: REQUEST_MASK.

Twin Design

Feasible Twins

Each S1 infeasible example is paired with a minimally different feasible example. The pair should flip the gold action from a non-edit action to EDIT while changing as little else as possible.

{
  "s1_infeasible": {
    "image_id": "img_no_car_001",
    "instruction": "Change the red sports car to blue.",
    "gold_action": "FLAG"
  },
  "s1_prime_feasible": {
    "image_id": "img_with_red_car_001",
    "instruction": "Change the red sports car to blue.",
    "gold_action": "EDIT"
  }
}

Counterfactual Twins

S3' creates a cacheable clarification test. The same image and vague instruction have two or more hidden intents, different edit plans, different success criteria, and cached direct-edit outputs. Blind direct editing should not satisfy both intents, creating a 50% direct-edit ceiling for paired binary cases.

Human Seed Slice

The human seed slice addresses the "synthetic data is unrealistic" criticism. It should collect free-form edit requests from 10-15 non-expert users, label feasibility and ambiguity with two independent annotators, report Cohen's kappa, and compare synthetic versus real request distributions using category and length statistics.
