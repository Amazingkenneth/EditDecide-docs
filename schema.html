JSON Specification

Schema and field rationale

The schema is broad because it is the join key across data construction, editor behavior, self-knowledge probes, judges, VOA decisions, and reproducibility after the compute freeze.

Design Principle

Every JSON field must answer at least one of four questions: what is the correct decision, why is it correct, which experiment or metric needs it, and can the result be reproduced after August 15?

Correct decision

decision_labels, feasibility, ambiguity, and intent_graph.

Reason

Evidence, gold reason, rationale type, and human annotations.

Experiment

tracks, rq_tags, paper_use, editor_runs, and self_knowledge.

Reproducibility

Dataset version, split, cache paths, provenance, pinned judge outputs.

FieldPurpose
sample_idStable join key across outputs, hidden states, decisions, judges, and figures.
dataset_versionPrevents metric drift across dataset releases.
splitPrevents leakage and supports reproducible train/dev/test evaluation.
twinsLinks feasible twins and counterfactual twins for SHR-ORR and twin-success experiments.
cacheStores paths for images, outputs, hidden states, predictions, and frozen artifacts.

This block separates the user's literal request from the system's structured interpretation. That separation is required for detecting false premises, ambiguity, and hallucinated edits.

Decision Labels Block

{
  "decision_labels": {
    "gold_action": "FLAG",
    "acceptable_actions": ["FLAG", "ABSTAIN"],
    "unacceptable_actions": ["EDIT"],
    "gold_reason": "No red sports car is visible in the image.",
    "decision_rationale_type": "visual_false_premise",
    "requires_user_input": false,
    "requires_mask": false
  }
}

The action space includes six actions, not only edit or refuse. This block defines what counts as correct, defensible, or clearly wrong under the benchmark cost model.

Feasibility Block

{
  "feasibility": {
    "label": "infeasible",
    "infeasible_type": "false_premise",
    "confidence": 1.0,
    "evidence": [
      {
        "evidence_id": "ev_001",
        "type": "object_absence",
        "text": "No vehicle is visible in the image.",
        "grounding": {
          "object_id": null,
          "bbox": null,
          "image_region": "entire_image"
        }
      }
    ],
    "minimal_fix": {
      "fix_type": "add_missing_object",
      "description": "Use an image containing a red sports car."
    }
  }
}

The feasibility block is the core G1 annotation. It provides the infeasibility label, evidence, visual grounding, and minimal repair needed to construct a feasible twin.
